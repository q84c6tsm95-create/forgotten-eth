Parity Multisig ~513,000 ETH
November 2017

A developer accidentally destroyed the Parity wallet library contract by calling kill() on it. This bricked 587 multisig wallets that depended on the library for all their logic, including withdrawals.

The affected wallets hold around 513,000 ETH. Multiple EIPs were proposed to recover the funds, but none gained enough community support. The ETH remains frozen.

Root cause: Delegatecall to a shared library. When the library self-destructed, every wallet that delegated to it became permanently non-functional.
AkuDreams NFT 11,539 ETH
April 2022

Two bugs in the Dutch auction contract for AkuDreams NFTs locked all proceeds permanently. A griefing attack exploited a refund loop, and a separate bug meant the contract owner could never withdraw either.

The 11,539 ETH sits in the contract with no function that can move it.

Root cause: Missing withdrawal function combined with a DoS vulnerability in the refund mechanism. The contract has no owner withdraw, no selfdestruct, and no fallback.
The DAO ~0 ETH (recovered via hard fork)
June 2016

The DAO raised 12.7M ETH in the largest crowdfund at the time. An attacker exploited a reentrancy bug to drain 3.6M ETH. The Ethereum community voted to hard fork the chain to reverse the hack.

The fork recovered the stolen funds, but it split Ethereum in two: ETH (the forked chain) and ETC (Ethereum Classic, the original chain where the hack stands).

Root cause: Reentrancy vulnerability. The withdrawal function sent ETH before updating the internal balance, letting the attacker call back in repeatedly.
GovernMental ~0 ETH (drained)
2016

A Ponzi scheme contract that collected 1,100 ETH. When the payout was triggered, the transaction ran out of gas because the recipient array was too large. A miner eventually extracted the ETH using a specially crafted transaction with extra gas.

Root cause: Unbounded array iteration. The contract tried to loop through all participants in a single transaction, exceeding the block gas limit.
HonestDice 206 ETH
2016

A gambling contract with 206 ETH stuck in it. The owner key still works, but the withdraw function references a variable that was shadowed by a local declaration. The owner can call it, but it always sends 0 ETH.

Root cause: Variable shadowing bug in Solidity. The withdraw function declared a local variable with the same name as the state variable, so it read 0 instead of the actual balance.
MoonCatRescue Genesis 100.8 ETH
2017

The original MoonCatRescue contract has a bug where adopted cats were sent to address(0) instead of the buyer. The 100.8 ETH in adoption fees is stuck because the owner withdrawal function also has a bug that prevents it from working.

Root cause: Two bugs. The adoption function used msg.sender in a context where it resolved to the zero address. The withdrawal function has a similar logic error.
The difference: Forgotten ETH tracks contracts where withdrawals still work. The ETH above is permanently stuck. If you have funds in a defunct contract, check if yours is recoverable.

Why does this happen?

Smart contracts on Ethereum are immutable. Once deployed, the code cannot be changed. If there's a bug in the withdrawal logic, or if a critical dependency is destroyed, the ETH stays locked forever. There is no admin, no support team, and no undo button.

This is by design. Immutability is what makes smart contracts trustworthy, but it also means mistakes are permanent. Modern contracts use upgradeable proxies and extensive auditing to mitigate this, but early Ethereum was more experimental.
